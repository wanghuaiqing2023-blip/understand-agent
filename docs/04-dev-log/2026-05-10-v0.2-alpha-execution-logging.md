# 开发日志：v0.2-alpha 执行轨迹日志

## 日期

2026-05-10

## 今天做了什么

- 新增 `TraceEvent` 和 `ExecutionLogger`。
- CLI 默认将执行轨迹写入 `.understand-agent/logs/*.jsonl`。
- CLI 执行类命令的 stdout 结果包含 `run_id`。
- CLI 执行结束后在 stderr 输出本次 trace 日志路径。
- 每次执行结束追加 `.understand-agent/logs/index.jsonl`。
- 新增 `logs list` 和 `logs show <run_id>` 查询命令。
- `ToolRegistry.run()` 记录工具调用开始和结束事件。
- 工具调用日志包含工具名、参数、权限等级、耗时、完整结果和原始错误。
- 新增日志相关单元测试和集成测试。

## 遇到的问题

- 日志不能污染 CLI stdout，否则会破坏工具调用结果的 JSON 契约。
- 日志可能包含文件内容和敏感信息，因此必须默认写入本地忽略目录。
- 如果不暴露本次日志路径，日志文件增多后会难以定位本次执行对应的 trace。
- 如果没有日志索引，用户只能靠文件名猜测最近执行，复盘成本会随日志数量增加。
- 如果 stdout 不包含 `run_id`，程序化调用者仍然需要解析 stderr 或索引文件才能关联执行结果和 trace。

## 关键决策

- 第一版日志用于失败归因证据保存，不做本地失败分类。
- 每次执行使用 `run_id` 串起完整事件流。
- 每个关键动作记录为一条 JSONL 事件，便于后续按时间线复盘。
- trace 路径输出到 stderr，保持 stdout 仍然只承载命令结果 JSON。
- stdout 顶层 `run_id` 是结果和日志之间的稳定关联字段。
- `logs` 查询命令本身不创建新的 trace，避免查看日志污染日志索引。

## 概念收获

- 失败归因的基础不是立刻判断“错在哪里”，而是先完整保存“发生了什么”。
- `TraceEvent` 是动作流转中的一个时间点，`run_id` 是把这些时间点串起来的执行轨迹。
- 日志系统会逐步成为评估、调试、错误恢复和记忆沉淀的共同地基。
