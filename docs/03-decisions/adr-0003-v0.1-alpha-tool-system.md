# ADR-0003：v0.1-alpha 工具注册系统起步方案

## 状态

accepted

## 背景

项目已经完成文档和概念地基，需要进入第一个可运行工程骨架。我们决定先实现工具注册系统，而不是直接实现完整自然语言 Agent。

这个选择服务于一个核心问题：工具如何被描述、注册、发现、调用，并把结果安全地返回给智能体系统。

## 决策

v0.1-alpha 使用 Python 3.13 实现顶层包 `understand_agent`，采用零外部依赖和标准库 `unittest`。

第一版包含：

- `ToolSpec`：描述工具名称、说明、输入 schema、权限等级和执行函数。
- `ToolResult`：统一工具调用结果。
- `ToolContext`：提供工作区根目录。
- `ToolRegistry`：注册、列出和调用工具。
- 只读工具：`list_files`、`read_file`、`search_text`。
- CLI：`python -m understand_agent tools` 和 `python -m understand_agent call ...`。
- GitHub Actions：自动运行单元测试和集成测试。

内部工具协议不绑定 OpenAI API 的 tool type。OpenAI 的 `type: "function"`、内置搜索工具、MCP 工具等未来应放在 adapter 层处理。

## 原因

Python 适合快速验证本地文件、CLI、测试和自动化。标准库测试可以避免项目早期被依赖安装和工具链配置分散注意力。

顶层包结构让仓库根目录下可以直接运行 `python -m understand_agent`，对学习和手动验证更直观。

工具协议保持模型无关，有利于未来接入 OpenAI、MCP、本地函数、Shell 或其他执行器。

## 替代方案

- 直接实现自然语言 Agent：更接近最终体验，但会把规划、模型调用、工具调用混在一起，不利于理解工具系统本身。
- 使用 Node.js / TypeScript：更适合后续产品化和异步编排，但第一轮本地工具验证成本更高。
- 引入 pytest：测试体验更好，但会增加早期依赖和安装步骤。

## 后果

项目从纯文档阶段进入可运行工程骨架。第一版工具能力只读、低风险、可测试，但还不具备自然语言任务理解、计划生成、写文件或命令执行能力。

后续新增工具时，应继续维护工具描述、权限级别、测试和文档同步。

## 复审条件

当项目开始接入 GPT-5.5、MCP、Shell 工具、写文件工具或浏览器工具时，需要复审工具协议是否要新增权限策略、调用日志、schema 校验和 adapter 层。
