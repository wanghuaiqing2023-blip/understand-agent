# ADR-0004：执行轨迹日志用于失败归因

## 状态

accepted

## 背景

随着项目从工具注册进入实际执行，后续跟踪智能体执行流程时会出现大量失败。为了判断任务为什么没有完成，项目需要保存可复盘的动作证据链。

本地端不应过早承担失败分类能力。第一版更重要的是完整记录事实，让人类或未来分析智能体可以基于日志做归因。

## 决策

引入执行轨迹日志系统：

- 每次 CLI 执行生成一个 `run_id`。
- 每个关键动作记录为一条 `TraceEvent`。
- 日志以 JSONL 写入 `.understand-agent/logs/{run_id}.jsonl`。
- 每次执行结束追加 `.understand-agent/logs/index.jsonl`，记录最近执行索引。
- 日志默认开启，但写入 `.gitignore` 忽略目录。
- stdout 保持 JSON 输出，并在执行类命令结果中包含 `run_id`。
- stderr 输出本次 trace 文件路径，方便人工快速定位日志。

第一版事件包括：

- `run_started`
- `cli_args_received`
- `registry_loaded`
- `tool_call_requested`
- `tool_call_started`
- `tool_call_finished`
- `run_finished`

工具调用日志尽量完整记录输入、输出、错误和耗时，但不做本地失败分类。

CLI 提供日志查询入口：

- `python -m understand_agent logs list`
- `python -m understand_agent logs show <run_id>`

## 原因

JSONL 简单、可审计、易于追加，也方便后续脚本或智能体读取分析。

把日志和 stdout 分离，可以同时满足机器可调用 CLI 与长期失败复盘。

索引文件让用户不需要记住每次执行的 `run_id`，可以从最近执行列表中找到需要复盘的 trace。

执行结果里的 `run_id` 让程序化调用者也能直接关联 stdout 结果和本地日志。

不做本地失败分类，可以避免早期用不成熟规则制造错误归因。当前阶段先保存证据，后续再基于真实失败案例设计分类体系。

## 替代方案

- 只输出控制台日志：实现简单，但无法长期追踪历史失败。
- 使用 SQLite：查询能力更强，但早期复杂度偏高。
- 立即设计完整失败分类：看似更智能，但缺少足够失败样本，容易过早抽象。

## 后果

每次 CLI 调用都会在本地留下执行轨迹。日志可能包含文件内容和敏感信息，因此必须保持在本地忽略目录，不进入 Git。

后续需要围绕日志增加查看、筛选、压缩和归因工具。

## 复审条件

当项目引入任务状态机、计划生成、模型调用或写文件/命令执行工具时，需要扩展事件类型，并重新评估是否引入失败分类、日志脱敏和日志查询能力。
