# 认知突破：通过行为流程学习智能体

## 触发

我们讨论了学习 Codex 这类工程型智能体时，是否应该直接阅读源码，还是先跟踪它完成真实任务时的执行流程。

新的判断是：在项目早期，直接阅读成熟智能体源码并不是最优学习方式。更有效的方式，是先观察它如何围绕目标行动，再把行动过程拆解成能力、概念和最小实现。

## 旧理解

一种直觉是：如果想实现接近 Codex 的智能体，就应该尽早研究 Codex 的源码或类似成熟系统的源码。

这种方式看似直接，但容易让学习者过早陷入成熟系统的复杂实现细节，包括模块划分、类结构、接口封装、历史兼容、性能优化和产品约束。

## 新理解

前期不读源码，只跟踪执行流程，是一种更明智的学习方式。

源码展示的是成熟系统的压缩结果，执行流程展示的是智能体如何把目标转化为行动。对于理解智能体本质而言，后者更接近第一性问题。

智能体的核心不是某个代码结构，而是一个行动闭环：

```text
目标
-> 理解边界
-> 搜集上下文
-> 形成计划
-> 选择工具
-> 执行动作
-> 观察反馈
-> 修正判断
-> 完成或继续
```

只有先看清这个闭环，后续的代码结构才有真实含义。

## 关键洞察

直接阅读成熟源码容易造成能力错觉：看懂了类、函数和模块，以为自己理解了智能体能力。

但代码结构只是能力的投影。真正需要掌握的是：

- 为什么智能体先读这个文件？
- 为什么它选择搜索而不是直接修改？
- 为什么某一步需要计划，某一步可以直接执行？
- 为什么测试失败后它怀疑这个方向？
- 为什么某些操作需要用户确认？
- 为什么总结时要说明风险和验证情况？

这些问题指向的是智能体的能力结构，而不是源码形式。

## 对项目的影响

本项目的学习路线应优先采用：

```text
观察行为
-> 抽象能力
-> 命名概念
-> 最小复现
-> 任务评估
-> 源码对照
```

成熟源码不是第一学习材料，而是后期对照材料。只有当我们已经拥有自己的概念模型、最小实现和真实问题之后，阅读成熟系统源码才会产生高质量收益。

这条路线有助于避免两类偏差：

- 形式逻辑偏差：把类、接口、模块误认为能力本身。
- 能力错觉偏差：把看懂别人执行过程误认为自己已经掌握能力。

## 后续行动

- 前期重点跟踪 Codex 完成真实任务时的执行流程。
- 每次观察都拆解为感知、规划、工具、执行、反馈、评估、记忆和协作等能力。
- 在 understand-agent 中优先实现最小能力闭环，而不是模仿成熟系统架构。
- 等项目拥有自己的 AgentLoop、ToolRegistry、TaskState、PermissionPolicy 等最小实现后，再阅读成熟源码进行对照。
- 可考虑新增 `docs/02-concepts/capability-illusion.md`，专门记录能力错觉及其规避方法。
